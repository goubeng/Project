# 业主方项目治理与风险驾驶舱 PRD（V1.0）

## 1. 产品背景与定位

### 1.1 背景

在政务平台建设、信息化项目、重点专项等场景中，传统项目管理系统以项目经理视角为主，侧重任务、进度与资源管理，难以满足业主方（甲方、监管方、领导层）对以下问题的核心诉求：

* 项目是否存在重大风险？
* 是否符合规范与审计要求？
* 是否有人负责并形成闭环？
* 是否可能留下治理与问责隐患？

### 1.2 产品定位

本系统定位为：
**以业主方视角为核心的项目治理与风险驾驶舱系统**。

核心不是“管理项目怎么干”，而是帮助业主方判断：

* 是否安全
* 是否可靠
* 是否可追责
* 是否可审计

### 1.3 核心价值

* 提前发现风险
* 强化履约与合规
* 明确责任与闭环
* 沉淀过程资产
* 支撑审计与问责

---

## 2. 目标用户与使用场景

### 2.1 目标用户

* 主管领导 / 分管领导
* 业主单位项目主管
* 监管与审计人员
* 平台建设统筹部门

### 2.2 核心使用场景

* 领导快速判断项目是否需要介入
* 业主方例会 / 汇报查看项目风险
* 审计、巡查、绩效考核支撑
* 项目验收与后评估决策支持

---

## 3. 产品设计原则（业主方视角）

* 风险优先，而非进度优先
* 异常优先，而非任务细节
* 责任优先，而非人员列表
* 治理证据优先，而非过程描述
* 自动总结优先，而非人工解读

---

## 4. 业主方核心判断模型

系统需支持业主方在 10 秒内回答以下 4 个核心问题：

1. 现在安全吗？（风险）
2. 交付靠谱吗？（质量）
3. 有没有人兜底？（责任）
4. 以后会不会留下坑？（合规与审计）

## 6. 核心功能模块

### 6.1 风险治理模块

#### 目标

让业主快速识别是否存在可能引发重大后果的风险。

#### 核心指标

* 未关闭高风险数量
* 红色风险数量
* 高风险超期数量
* 风险关闭率

#### 功能

* 高风险自动标红
* 风险超期自动升级
* 风险趋势（上升 / 下降）

---

### 6.2 交付质量与验收模块

#### 目标

评估项目是否稳定交付，是否存在反复整改或质量不稳定。

#### 核心指标

* 一次验收通过率
* 平均整改次数
* 未关闭质量问题数
* 重大质量问题数

#### 功能

* 多次整改自动预警
* 重大质量问题自动标红
* 验收稳定性趋势

---

### 6.3 责任体系与执行力模块

#### 目标

确保项目关键事项有人负责、有反馈、有闭环。

#### 核心指标

* 关键责任人到位率
* 督办事项未闭环数
* 超期任务数
* 问题平均响应时长

#### 功能

* 关键任务责任到人
* 督办超期自动预警
* 问题响应时效统计

---

### 6.4 合规与治理证据模块

#### 目标

确保项目过程可审计、可追溯、可交接。

#### 核心指标

* 阶段成果齐套率
* 验收资料完整率
* 关键决策留痕率
* 审计问题数量

#### 功能

* 必备文档清单校验
* 阶段成果缺失预警
* 关键审批与评审留痕

---

## 7. 业主方治理健康指数模型（示例）

总分 100 分，构成如下：

1. 风险安全（30 分）

* 无高风险：30
* 1 个高风险：15
* ≥2 个高风险：0

2. 交付质量稳定性（25 分）

* 一次验收通过率 ≥90%：25
* 存在返工：12
* 多次整改：0

3. 责任闭环（20 分）

* 督办全部闭环：20
* 存在未闭环：10
* 多项未闭环：0

4. 合规与资料完备（15 分）

* 阶段资料齐套：15
* 存在缺失：8
* 多项缺失：0

5. 执行与响应（10 分）

* 问题及时响应：10
* 存在响应超期：5
* 多项严重超期：0

输出：

* 治理健康指数
* 状态灯（绿 / 黄 / 红）

## 14. 领导汇报一页式价值总览（One-Pager）

### 一句话定位

本系统是：
**为业主方和领导层提供“是否介入、为何介入、如何追责”的项目治理与风险判断平台。**

### 四个核心价值（领导视角）

1. 提前看到风险，而不是事后追责
2. 看清责任主体，而不是模糊推诿
3. 形成治理证据，而不是口头汇报
4. 支撑审计问责，而不是临时补材料

### 领导最关心的四个问题与系统支撑

| 领导问题    | 系统回答方式       |
| ------- | ------------ |
| 会不会出事？  | 风险状态灯 + 红色预警 |
| 靠不靠谱？   | 验收稳定性 + 质量趋势 |
| 谁来负责？   | 责任到人 + 督办闭环  |
| 会不会被追责？ | 合规留痕 + 审计支撑  |

## 15. 单项目业主驾驶舱原型级结构（线框说明）

### 页面整体布局（从上到下）

#### A. 顶部决策条（Decision Bar）

* 项目名称
* 当前阶段
* 状态灯（绿/黄/红）
* 治理健康指数
* 是否建议领导介入（高亮提示）

用途：3 秒内判断是否需要关注

---

#### B. 风险与安全区（Risk Zone）

* 未关闭高风险数量（大号数字）
* 红色预警数量
* 最近新增高风险
* 高风险超期数

用途：判断是否存在“可能出事”的风险

---

#### C. 交付质量区（Quality Zone）

* 一次验收通过率（进度环）
* 平均整改次数（趋势）
* 未关闭重大质量问题
* 质量问题变化趋势（↑ ↓）

用途：判断项目是否稳定、是否反复整改

---

#### D. 责任与闭环区（Accountability Zone）

* 督办未闭环数（红色强调）
* 超期任务数
* 关键责任人是否到位（是/否）
* 问题平均响应时长

用途：判断是否“有人兜底”

---

#### E. 合规与证据区（Governance Zone）

* 阶段成果齐套率
* 验收资料完整率
* 关键审批留痕完整度
* 审计问题数量

用途：判断是否“留坑”与是否可审计

---

#### F. 异常画像与介入依据区（Leadership Insight）

* 自动异常总结（自然语言）
* 最近 3 条关键异常
* 最近 3 条领导督办

用途：给领导“为什么要介入”的直接理由

## 17. 与原项目监管系统的边界说明

本系统与传统 PMO 项目管理系统边界如下：

* PMO 系统：

  * 管任务、管计划、管资源
  * 面向项目经理与执行人员

* 本业主治理系统：

  * 管风险、管责任、管合规、管审计
  * 面向领导层与业主监管方

两套系统定位互补，但本系统为治理与监管主系统。

## 20. 原型设计关键 UI 原则（给设计师 / 原型师）

1. 红色优先级最高（风险 > 质量 > 责任 > 合规）
2. 所有核心卡片支持“一键展开明细”
3. 默认隐藏复杂表格，避免领导信息过载
4. 自动总结区域必须视觉突出
5. 是否介入建议必须比任何 KPI 更醒目

## 21. ASCII 原型中各核心指标的业务含义与领导解读说明（新增）

### A. 顶部决策条（Decision Bar）

| 指标         | 含义                | 领导视角解读          |
| ---------- | ----------------- | --------------- |
| 状态灯（绿/黄/红） | 基于治理健康指数与红色规则综合判定 | 一眼判断项目是否存在治理风险  |
| 治理健康指数     | 对风险、质量、责任、合规的加权评分 | 项目整体是否“健康”      |
| 是否建议介入     | 规则引擎自动判断          | 系统是否建议领导关注或批示   |
| 红色预警数量     | 当前触发红色规则的事项数量     | 是否存在可能引发严重后果的问题 |

---

### B. 风险与安全区（Risk Zone）

| 指标     | 管理含义       | 领导关心点         |
| ------ | ---------- | ------------- |
| 未关闭高风险 | 尚未关闭的高等级风险 | 是否存在可能“出事”的隐患 |
| 红色预警   | 已触发强制关注规则  | 是否需要立即干预      |
| 高风险超期  | 高风险未按时关闭   | 风险是否被放任       |
| 风险趋势   | 风险数量变化     | 项目在变好还是变差     |

---

### C. 交付质量区（Quality Zone）

| 指标      | 管理含义      | 领导关心点     |
| ------- | --------- | --------- |

| 平均整改次数  | 每个阶段平均返工  | 是否反复整改    |
| 重大质量问题  | 影响整体交付的问题 | 是否存在系统性缺陷 |
| 质量趋势    | 质量问题变化    | 管理是否有效    |

---

| 责任人到位 | 关键岗位是否明确   | 是否有人兜底  |

| 内容     | 作用       | 领导价值       |
| ------ | -------- | ---------- |
| 自动异常总结 | 多指标合成一句话 | 减少领导解读成本   |
| 最近关键异常 | 当前最严重问题  | 快速定位问题     |
| 最近领导督办 | 批示与整改状态  | 判断是否需要再次介入 |

## 22. 现有领导侧页面向业主治理驾驶舱改造方法论（新增）

### 22.1 判断是否覆盖领导关注点的检查表

若页面无法快速回答以下问题，则不合格：

1. 现在这个项目会不会出事？
2. 交付到底稳不稳？
3. 有没有人兜底？
4. 将来会不会被审计追责？
5. 我现在要不要介入？

---

### 22.2 从“项目管理页”改造成“业主治理页”的高效改造步骤

#### Step 1：删除

* 删除详细任务列表
* 删除甘特图作为主视图
* 删除资源分配与工时类信息

#### Step 2：升级

* 将进度完成率 → 升级为【验收稳定性 + 返工率】
* 将问题列表 → 升级为【红色预警 + 高风险】
* 将人员列表 → 升级为【关键责任是否到位】

#### Step 3：新增（业主专属）

* 是否建议领导介入（系统判断）
* 异常画像自动总结
* 合规与资料齐套率
* 督办闭环率

#### Step 4：重构布局

* 顶部必须是：状态 + 介入建议
* 中部必须是：风险 / 质量 / 责任 / 合规
* 底部必须是：介入依据（异常画像）

## 23. 领导侧页面 vs 业主治理驾驶舱对比（新增）

| 维度   | 传统领导驾驶舱  | 本业主治理驾驶舱 |
| ---- | -------- | -------- |
| 关注点  | 进度、投入    | 风险、责任、合规 |
| 核心问题 | 干到哪了     | 会不会出事    |
| 页面结构 | KPI + 图表 | 状态 + 异常  |
| 决策支撑 | 需要解读     | 系统给结论    |
| 审计支撑 | 弱        | 强        |

（新增内容，用于指导原型与现有系统的业主侧重构）

1. 红色优先级最高（风险 > 质量 > 责任 > 合规）
2. 所有核心卡片支持“一键展开明细”
3. 默认隐藏复杂表格，避免领导信息过载
4. 自动总结区域必须视觉突出
5. 是否介入建议必须比任何 KPI 更醒目
